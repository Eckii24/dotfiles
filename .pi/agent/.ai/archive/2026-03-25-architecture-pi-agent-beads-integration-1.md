# Introduction

![Status: Deprecated](https://img.shields.io/badge/status-Deprecated-red)

> Superseded by markdown-first current-work workflow documented in `.ai/archive/2026-03-26-process-current-work-workflow-plan.md`.

This plan defines a low-risk, phased integration of plain Beads (`bd`) into the current `~/.pi/agent` setup. The target state is: Beads becomes the operational source of truth for work intake, prioritization, dependencies, claiming, status, and completion, while `.ai/` remains the source of truth for long-form markdown artifacts generated by the agent workflows (specs, implementation plans, review notes, and progress files). BeadHub is explicitly out of scope.

## Source of Truth Boundary

| Concern | Source of Truth | Storage / Surface | Notes |
|---|---|---|---|
| Backlog, ready work, dependencies, assignee/claim state, workflow status | Beads | `.beads/` via `bd` CLI | This is the canonical operational queue |
| Specs, implementation plans, review outputs, progress logs, handoff notes | Markdown artifacts | `.ai/*.md` | These remain file-based and reviewable |
| Thin role definitions | Markdown agent prompts | `agents/*.md` | Keep thin; do not embed full Beads docs here |
| Orchestration workflows | Markdown prompt templates | `prompts/*.md` | Prompts coordinate Beads + `.ai/` artifact generation |
| User clarifications / approvals | `questionnaire` flow + session history | session + linked `.ai/` artifact notes when durable | Do not replace questionnaire with Beads comments |
| Sub-agent delegation | `subagent` tool | prompt workflows | Beads tracks the work item; prompts still delegate |
| Fresh-agent task cadence | Prompt workflows + bead structure | epic/story entrypoints + child beads | Use an epic-level Ralph loop: each iteration selects one ready child task, runs it in a fresh agent context, persists results, then stops before the next iteration |

## 1. Requirements & Constraints

- **REQ-001**: Integrate plain Beads (`bd`) into `~/.pi/agent` as the operational workflow tracker.
- **REQ-002**: Explicitly exclude BeadHub, hosted sync flows, and any BeadHub-specific assumptions.
- **REQ-003**: Preserve the current architecture: thin role agents in `agents/*.md`, orchestration prompts in `prompts/*.md`, and markdown artifacts in `.ai/`.
- **REQ-004**: Keep `.ai/` as the location for plans, specs, reviews, progress files, and other markdown artifacts created by agents.
- **REQ-005**: Make Beads the source of truth for work state, dependencies, work selection, claim/ownership, and completion state.
- **REQ-006**: Preserve `questionnaire` as the user-facing clarification and approval mechanism.
- **REQ-007**: Preserve `subagent` as the delegation mechanism for substantial work.
- **REQ-008**: Keep `AGENTS.md` rules authoritative and augment them with Beads guidance rather than replacing them.
- **REQ-009**: Roll out in low-risk phases with additive changes first, then optional automation.
- **REQ-010**: Support a migration path from the current `.ai/`-only workflow habits to Beads-tracked operational work.
- **REQ-011**: Ensure every implementation phase has deterministic eval gates.
- **REQ-012**: Maintain a clear current-progress / handoff section in `.ai/` so future agents can resume work.
- **REQ-013**: Bake in a fresh-agent cadence policy: each epic/story workflow must decompose into child tasks that can be claimed and completed one at a time by fresh agents/sessions.
- **REQ-014**: Each substantial subagent invocation must handle exactly one claimed child task/bead, update Beads and linked `.ai/` artifacts, and then exit cleanly.
- **REQ-015**: Prompt templates must explicitly invoke `bd prime --stealth`; no separate Beads session-context extension is allowed.
- **REQ-017**: This repo must keep Beads in local-only stealth mode (`no-git-ops: true`) and must not rely on git sync/remotes or tracked `.beads/` data.
- **REQ-016**: `extensions/ralph-loop.ts` must default to a fresh Pi session/context per iteration so Ralph-loop runs do not accumulate the same active chat history across iterations.
- **REQ-018**: `/story-to-code` must create the top-level bead as `ado-<id>` when it resolves a numeric Azure DevOps work item ID, and child beads beneath that story must inherit IDs like `ado-<id>.1`; all other workflows keep the normal `agent` prefix.
- **CON-001**: The current repo is a Pi agent configuration workspace, not a typical application repo; validation must rely on file checks, prompt behavior, and `bd` health commands rather than app builds/tests.
- **CON-002**: Existing prompt templates and agent files are intentionally lean; avoid turning them into large workflow manuals.
- **CON-003**: Existing custom extensions (`extensions/*.ts`) already participate in startup/runtime behavior; any Beads automation must remain additive and non-breaking.
- **CON-004**: Existing `.ai/*.md` files should be treated as historical artifacts unless explicitly migrated into active Beads-linked workflows.
- **CON-005**: Without BeadHub, fresh top-level orchestrator restarts can be encouraged and documented, but only subagent freshness can be reliably enforced inside prompt workflows.
- **CON-006**: `extensions/ralph-loop.ts` now resets to a fresh Pi session/context per iteration by default, while preserving `--same-session` as an explicit legacy escape hatch.
- **CON-007**: For this workflow, a fresh Pi session/context per Ralph iteration is sufficient; a brand-new external `pi` process is not required if the active context is actually reset.
- **GUD-001**: Put high-detail Beads workflow guidance in `.beads/PRIME.md`, not duplicated across every agent file.
- **GUD-002**: Prefer prompt-level orchestration updates before adding automatic session hooks.
- **GUD-003**: Keep Beads ↔ `.ai/` linkage explicit by storing artifact file paths in bead comments and/or metadata.
- **GUD-004**: Prefer an epic-level Ralph loop: each iteration should select one ready child task, run it in a fresh context, persist results, and stop before selecting the next task.
- **PAT-001**: Use Beads as the operational SSOT and `.ai/` as the artifact SSOT.
- **PAT-002**: Preserve the current skill-first approach: skills remain the deep instruction source; prompts and agents stay thin.
- **PAT-003**: Minimize settings churn; do not touch `settings.json` unless Beads integration later proves it is required.
- **PAT-004**: Model each top-level workflow as an epic/story bead with child phase/task beads; use one fresh agent/session per child bead rather than letting one agent span multiple child tasks.

## 3. Alternatives

- **ALT-001**: Keep `.ai/` as the operational source of truth and mirror state into Beads. **Rejected** because it duplicates status/dependency state and guarantees drift.
- **ALT-002**: Move specs and plans into Beads and stop generating `.ai/` markdown artifacts. **Rejected** because it conflicts with the current architecture and removes durable file-based handoff artifacts.
- **ALT-003**: Put all Beads workflow rules directly into every agent file. **Rejected** because it breaks the thin-agent pattern and creates duplicated instructions.
- **ALT-004**: Add BeadHub or remote/federated features now. **Rejected** because the requested scope is plain Beads only.
- **ALT-005**: Add automatic hooks/extensions first and skip prompt-level migration. **Rejected** because it increases risk and makes failures harder to diagnose.
- **ALT-006**: Treat same-session Ralph-loop behavior as sufficient without adding a fresh-session reset. **Rejected** because this workflow needs a fresh context window per iteration, which is now enforced explicitly in `extensions/ralph-loop.ts`.

## 8. Risks & Assumptions

- **RISK-001**: `bd init` or Dolt prerequisites may add bootstrap friction in a repo that currently has no Beads state.
- **RISK-002**: If Beads instructions are duplicated across `AGENTS.md`, prompts, agents, and extensions, the integration will drift and become noisy.
- **RISK-003**: Over-automating `bd prime` injection too early may consume unnecessary context tokens or create surprising session behavior.
- **RISK-004**: During migration, users or agents may continue maintaining markdown TODO lists, causing drift between Beads and `.ai/`.
- **RISK-005**: Updating prompt templates without preserving approval gates could weaken the current questionnaire-driven workflow.
- **RISK-006**: If prompt-level `bd prime` usage proves too manual, later sessions may still want a lightweight runtime helper; that should be evaluated after the prompt-driven pilot, not before.
- **RISK-007**: If Ralph-loop fresh-session resets fail to seed enough durable context, iterations may repeatedly rediscover the same work or stall.
- **RISK-008**: If the new default surprises users who relied on same-session accumulation, they may need the explicit `--same-session` escape hatch.
- **ASSUMPTION-001**: Plain `bd` CLI usage is sufficient for this integration; no BeadHub, MCP, or remote service layer is required.
- **ASSUMPTION-002**: `.ai/` should continue to store specs, plans, reviews, and rollout notes rather than moving those artifacts into Beads bodies.
- **ASSUMPTION-003**: Existing `.ai/*.md` files are mostly historical and do not need bulk import into Beads; only active workflows need linking.
- **ASSUMPTION-004**: Minimal agent-file edits are acceptable as long as the thin-agent pattern is preserved and Beads workflow details stay centralized elsewhere.
- **ASSUMPTION-005**: Validation for this repo will rely primarily on deterministic file checks and `bd` CLI smoke tests, not traditional build/test pipelines.
- **ASSUMPTION-006**: The confirmed Beads prefix for this repo is `agent`.
- **ASSUMPTION-007**: The confirmed rollout mode is prompt-driven only for this iteration; no Beads session-context extension will be added now.
- **ASSUMPTION-008**: The first rollout slice will rely on prompt-driven `bd prime --stealth`, one-task subagent invocations, and a Ralph loop that resets to a fresh Pi session/context per iteration by default.
- **ASSUMPTION-009**: Using Pi's supported session APIs (`ctx.newSession(...)`) is an acceptable way to achieve a fresh context window without launching a brand-new external `pi` process.
- **ASSUMPTION-010**: If fresh-session automation is revisited later, it should remain compatible with the prompt-driven Beads workflow and the pilot validation path.
